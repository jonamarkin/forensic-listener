\section{Executive Summary}

Forensic Listener is a database-centric Ethereum forensic investigation system.
The project was built to answer a database systems question:

\begin{quote}
How should Ethereum forensic data be modeled when the application must support
relational queries, graph traversal, and similarity search?
\end{quote}

The final design uses:

\begin{itemize}
\item \textbf{PostgreSQL} for transactions, accounts, flags, notes, tags,
cases, and contract metadata
\item \textbf{Neo4j} for multi-hop tracing, hub discovery, and circular-flow
analysis
\item \textbf{pgvector} inside PostgreSQL for contract and behavior similarity
\end{itemize}

The implementation consists of a Go backend for ingestion and API delivery, and
a Next.js frontend for analyst workflows. The current system supports account
dossiers, alert triage, graph tracing, contract similarity, transaction
inspection, and case management.

The system ingests live Ethereum transaction data through a WebSocket feed. In
addition, it bootstraps a small curated known-entity reference set so that
well-known tokens, routers, and hubs can be labeled from the beginning of an
investigation. This seeded reference layer improves interpretability, but it
does not replace live blockchain ingestion.

\section{Design Decisions and Rationale}

\subsection{PostgreSQL as the Source of Truth}

PostgreSQL was chosen as the main data store because the project requires
transactional integrity, constraints, foreign keys, and efficient relational
queries over accounts, transactions, flags, notes, tags, and cases. This makes
it the strongest fit for the core ledger and workflow data.

\textbf{Trade-off:} Some graph-shaped questions are less natural in SQL, which
motivated the use of Neo4j for graph-native analysis.

\subsection{Neo4j for Graph-Native Queries}

Neo4j was added because the project needs multi-hop tracing, hub analysis, and
circular-flow detection. These queries are naturally expressed as graph
traversals and are easier to reason about in a graph database than in a purely
relational model.

\textbf{Trade-off:} Introducing Neo4j increases deployment and synchronization
complexity because the system now depends on multiple databases.

\subsection{pgvector Inside PostgreSQL}

pgvector was used to support similarity search for contract bytecode and
behavior profiles without introducing a separate vector service. Keeping vector
search inside PostgreSQL makes joins with relational metadata simpler and keeps
the overall architecture more compact.

\textbf{Trade-off:} pgvector is less specialized than a dedicated vector
database, but it is a better fit for a compact academic system.

\subsection{JSONB Instead of MongoDB}

The project stores some semi-structured data, especially ABI and behavior
feature maps. These were modeled with \texttt{JSONB} inside PostgreSQL because
the semi-structured requirements were limited and still closely tied to
relational entities.

\textbf{Trade-off:} PostgreSQL carries both relational and limited
semi-structured data, but this keeps the system simpler than introducing a
fourth datastore.

\subsection{Separate Backend and Frontend}

The Go backend and Next.js frontend were kept as separate components so that
ingestion, data access, and API logic remain isolated from presentation logic.
This separation also makes it easier to evolve the analyst interface without
mixing it into ingestion and database code.

\textbf{Trade-off:} The final deployment has two application runtimes rather
than one monolithic service.

\section{Conclusion}

Forensic Listener is a database systems project built around a real forensic use
case. Its main contribution is showing how relational, graph, and vector models
can be combined in one application:

\begin{itemize}
\item PostgreSQL answers structured ledger and workflow questions
\item Neo4j answers multi-hop connection questions
\item pgvector answers similarity questions
\end{itemize}

The system is already usable as a prototype for analyst workflows, and it also
demonstrates the database design decisions, trade-offs, and schema modeling
expected in a database course project.
